# Core Principles

## TOP_LEVEL_PRINCIPLE

**「ユーザーの意図を完遂させる」**

- すべてのフェーズ、すべての判断、すべてのレビューはこの原則に帰結する
- 判断が衝突した場合は「どちらがよりユーザーの意図の完遂に貢献するか」で優先度を決定する
- この原則は例外なく最優先であり、いかなる技術的判断・プロセス判断よりも上位に位置する

## STRUCTURAL_PRINCIPLES（構造原則）

### SP-1: オペレーターは意思決定者である

- オペレーター = 作業者ではなく「判断・承認・差し戻し」の責任者
- 従来の開発チームにおけるテックリード・スクラムマスター・PM・企画者の判断責務を1人に集約した存在
- AIの出力を無条件で受け入れることは決してしない
- エンジニア経験3年以上を前提とする（AIの出力の妥当性を評価できる能力が必要）

**オペレーターがやること:**
- 顧客の真の課題の見極め
- 技術選定の最終判断
- フィードバックの優先度付け
- AIレビュー結果サマリーの妥当性判断

**オペレーターがやらないこと:**
- 実装の主体にはならない（→ コーディングエージェント）
- 監査の主体にはならない（→ システム監査官）

### SP-2: 5つのAIロールは統合しない

ロール分離は効率的な分業のためではなく、**相互牽制による品質保証のための構造設計**である。

| ロール | 責務 | 主要稼働フェーズ |
|--------|------|-----------------|
| PM・スクラムマスター | 進捗管理、要件整合性確認、フィードバック共有 | Phase 7-8 |
| コーディングエージェント | 実装の主体 | Phase 5以降 |
| コードレビュアー | 7つのレビュー視点に基づくコード品質レビュー（品質ゲート） | Phase 7-8 |
| システム監査官 | 安全性・安定性・可用性に特化した独立監査（安全ゲート） | Phase 7-8 |
| ユーザー・運用サポート | ペルソナベースのテストシナリオ策定・UX検証 | Phase 6以降 |

**分離を維持する3つの理由:**
1. **異質性の担保:** 同一AIが実装と監査を兼ねると、自身の出力に対する批判的検証が構造的に弱化する
2. **相互牽制の実効性:** 牽制は独立性があって初めて機能する。利益相反は牽制を形骸化させる
3. **盲点の非重複:** 異なるロールは異なる盲点を持つ。分離により単一視点では検出できない問題を構造的に捕捉する

**分離が必要な具体的理由:**
- PM vs コードレビュアー: PMは「進捗を前に進める」、レビュアーは「品質基準に達するまで止める」→ 本質的に相反する責務
- コードレビュアー vs システム監査官: 「きれいか・保守しやすいか」vs「安全か・壊れないか」→ 兼任するとどちらかのレンズが支配的になる

### SP-3: レビューは2つの独立したゲートを通過させる

```
コーディングエージェント → 実装を出力
    ↓
コードレビュアー → 品質ゲート（7つのレビュー視点）
    ↓
システム監査官 → 安全ゲート（安全性・安定性・可用性）
    ↓
オペレーター → 両方のサマリーを読み、承認 or 差し戻しを判断
```

- どちらか一方のゲートの省略は許可しない
- オペレーターはコードを逐行レビューするのではなく、AIのレビュー結果サマリーの妥当性を判断する

### SP-4: 壁打ちの姿勢はフェーズで変える

| フェーズ | 優位性 | 姿勢 | 理由 |
|---------|--------|------|------|
| Phase 0-2 | オペレーター優位 | 引き出す | オペレーターが持つドメイン知識（業界特性、既存システム、実業務の癖）をAIが引き出す |
| Phase 3-5 | AI優位 | 提案する | AIの技術的知見を活かし選択肢を提示。ただし最終判断はオペレーター（ドメイン知識が判断を左右する） |

## IMPLEMENTATION_QUALITY_PRINCIPLES（実装品質原則）

### IQ-1: 汎用化はユーザー価値に貢献する場合のみ
- 開発効率性・システム拡張性・可用性に貢献がある場合 → パフォーマンスの軽微な低下を許容して積極的に採用
- 用途のない抽象化は可読性を落とす → 入れない

### IQ-2: 他者が読んで意図がわかるか
- コード、設計、ドキュメントすべてにおいて「配慮」を求める
- マジックナンバー、意味のない変数名、意図不明なフォルダ構成は「配慮の欠如」

### IQ-3: レビュー前の前提条件
- エラーハンドリングの設計はレビュー提出前に完了していること
- 分岐の将来性（種類が増えるかどうか）はレビュー前に確認済みであること
- 「とりあえず動く」状態でレビューに出さない

## CHECK_AND_BALANCE（牽制関係マップ）

| 牽制の方向 | 内容 |
|-----------|------|
| コーディングエージェント → コードレビュアー | 実装の出力を7つのレビュー視点でコード品質検証（品質ゲート） |
| コーディングエージェント → システム監査官 | 実装の出力を安全性・安定性・可用性で独立検証（安全ゲート） |
| PM・スクラムマスター → コーディングエージェント | 要件との整合性、進捗の妥当性を確認し逸脱を検知 |
| ユーザー・運用サポート → PM・スクラムマスター | PMが定義した要件・優先度をエンドユーザー視点で検証 |
| システム監査官 → 全ロール | セキュリティ・リソースコストの指摘は全ロールに対して最優先で適用 |

## OPERATOR_REVIEW_CHECKPOINTS（オペレーターのレビュー判断ポイント）

1. **7視点すべてがサマリーに含まれているか** — 視点の欠落がないか確認（「パフォーマンスに言及なし」＝未検証の可能性）
2. **各視点の判定根拠が具体的か** — 「問題なし」だけでは不十分。何を確認して問題なしと判断したかが明示されているか
3. **指摘事項の重大度判定が妥当か** — セキュリティ・リソースコストの問題が軽微扱いされていないか
4. **ドメイン知識から見て見落としがないか** — 業界固有の制約、レガシーシステムとの互換性等
