## 最も重要なフェーズ

AIネイティブ開発において、Phase 0〜2は最も重要なフェーズだ。

理由は単純だ。AIは実装が速い。驚くほど速い。だからこそ、**間違ったものを高速に作ることほど有害なことはない**。

Phase 0〜2は「何を作るか」を見極めるフェーズだ。ここでの判断を間違えると、以降のすべてのフェーズが無駄になる。逆に、ここで顧客の真の課題を正確に捉えられれば、Phase 3以降はAIの速度を最大限に活かして高速に進められる。

## Phase 0：現状把握と目標設定

### 目的

顧客業務の現状と理想の姿、直近で目指す目標地点を設定する。

### 何をするか

4つの情報を整理する。

1. **顧客の業界と事業内容** ── 相手が何者かを理解する
2. **顧客業務の現状** ── 今どうやっているかを具体的に把握する
3. **理想の姿** ── 最終的にどうなりたいかを描く
4. **直近の目標地点** ── まず何を達成するかを決める

### なぜこの4項目なのか

多くのプロジェクトが失敗する原因の1つは、「直近の目標」と「最終的な理想」の区別がつかないまま走り出すことだ。

理想は大きい。だが、それを一度に実現しようとすると複雑性が爆発する。直近の目標を明確に切り出し、そこに集中することで、現実的なスコープでプロジェクトを進められる。

### 壁打ちの姿勢：引き出す

このフェーズでのAI（ナビゲーター）の姿勢は「引き出す」だ。

オペレーターは業界特性、既存システム、実業務の癖といったドメイン知識を持っている。AIにはこの知識がない。だからAIは答えを与えるのではなく、問いを投げてオペレーターの思考を引き出す。

- 「その業界では一般的にどんな課題がありますか？」
- 「現状」と「理想」のギャップは具体的に何ですか？」
- 「直近の目標は現実的ですか？なぜそう判断しましたか？」

オペレーターが「なんとなくこう思う」で済ませようとしたら、「なぜそう判断したのか」を問い、根拠を言語化させる。これがPhase 0〜2を通じた基本姿勢だ。

## Phase 1：真の課題の追求

### 目的

顧客の表面的な要望の裏にある真の課題を追求する。

### 「要望」と「課題」は違う

顧客が「こういう機能がほしい」と言うとき、それは要望であって課題ではない。

例を挙げよう。顧客が「日報を自動集計する機能がほしい」と言ったとする。要望通りに日報集計機能を作ることはできる。だが、なぜ日報を集計したいのか。掘り下げると「プロジェクトごとの工数が把握できていない」という課題が出てくるかもしれない。さらに掘ると「赤字プロジェクトを早期に検知したい」という真の課題に行き着くかもしれない。

真の課題が「赤字プロジェクトの早期検知」なら、日報集計は1つの手段に過ぎない。もっと効果的な手段があるかもしれない。

### 5 Whysで掘り下げる

「なぜそれが問題なのか？」を繰り返し掘り下げる。

:::details 5 Whysの具体例
- なぜ日報を集計したいのか → プロジェクトごとの工数を知りたい
- なぜ工数を知りたいのか → 利益率を把握したい
- なぜ利益率を把握したいのか → 赤字プロジェクトを早期に止めたい
- なぜ早期に止めたいのか → 過去に赤字が膨らんで大損した経験がある

ここまで掘ると、「日報集計」ではなく「プロジェクト損益のリアルタイム可視化」が本当に必要な機能だとわかる。
:::

### 完了条件：1文で言語化

Phase 1の完了条件は厳格だ。

:::message
**Phase 1 完了条件：** 顧客の真の課題が1文で言語化できていること。「業務を効率化する」のような抽象表現は認めない。具体的で、顧客の業務文脈に根差していて、Phase 0の現状と理想のギャップを反映している1文を求める。
:::

「業務を効率化する」のような抽象表現は認めない。「月次の棚卸作業で発生する手入力ミス（年間約200件）を構造的にゼロにする」のように、具体的で、顧客の業務文脈に根差していて、Phase 0の現状と理想のギャップを反映している1文を求める。

この1文が言えない状態でPhase 2に進むことは許可しない。

## Phase 2：業務と実運用の明確化

### 目的

ユーザーの業務と実運用について、システムで解決すべき課題を明確化する。

### ユースケースを具体的なシナリオで出す

Phase 1で課題を特定したら、次はそれを「誰が、いつ、どこで、何をするか」のレベルで具体化する。

抽象的なユースケース：
> 「ユーザーが在庫を確認する」

具体的なユースケース：
> 「倉庫担当の田中さんが、毎朝8時に事務所のPCで在庫一覧画面を開き、前日の出荷分が反映されているか確認する。不一致があれば出荷伝票と突合し、差異を報告する」

この具体性の違いが、後のI/F設計やUI設計の品質を決定的に左右する。

### システム化の仕分け

すべてをシステム化すればいいわけではない。

- 頻度が低く、判断が必要な作業はシステム化しない方が良い場合がある
- 既存の運用でうまく回っている部分を無理にシステム化すると、かえって混乱を招く
- 法規制や業界慣行で紙の帳票が必要な場合もある

「システムで解決すべき課題」と「システム外で対応すべき課題」を仕分けることが、Phase 2の重要な作業の1つだ。

### 完了条件

- ユースケースが具体的なシナリオとして3つ以上出ている
- 各ユースケースでユーザーの操作フロー（入力→処理→出力）が明確である
- システムで解決すべき課題と、システム外で対応すべき課題が仕分けられている

## Phase 0〜2 総合ゲート

Phase 3に進む前に、以下の2条件を必ず確認する。

:::message
**Phase 0〜2 総合ゲート条件：**
- **ゲート条件1：** 顧客の真の課題が1文で言語化できている
- **ゲート条件2：** ユースケースが具体的なシナリオとして3つ以上出ている

この2条件を**同時に満たさない限り、Phase 3には進めない**。
:::

これは意図的に厳しくしている。Phase 3以降は技術的な検討に入るため、AIの提案力が活きる領域だ。だからこそ、「何を作るか」が曖昧なまま技術的検討に入ることを構造的に防ぐ必要がある。

ゲートを甘くすることは簡単だ。「だいたいわかったから進もう」と言えばいい。だが、その「だいたい」が後工程で手戻りとして返ってくる。AIの実装速度が速い分、手戻りの規模も大きくなる。

だからここは妥協しない。
