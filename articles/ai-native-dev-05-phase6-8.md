## Phase 6：プロトタイプベースのフィードバック

### コードではなく「動く画面」で対話する

Phase 5で作ったプロトタイプを、業務に近い人（営業・顧客・エンドユーザー）に見せる。

ここでの対話手段は、要件定義書でも画面設計書でもない。**動くプロトタイプ**だ。

なぜか。紙のドキュメントや静的な画面設計では、業務の人は「正しく想像する」ことができない。「この画面で在庫を確認できます」と書かれた設計書を見ても、実際に操作してみないと「使いやすいか」「業務フローに合っているか」は判断できない。

動く画面を見せて初めて、「ここで前日の出荷分がすぐ見えないと困る」「この画面遷移は日常業務だと面倒」といった具体的なフィードバックが出てくる。

### フィードバック収集の技術

フィードバックの質は、聞き方で決まる。

:::details フィードバック収集の「良い聞き方・悪い聞き方」
**悪い聞き方：**「何か違和感がありますか？」
→ 「特にないです」で終わる。人間は違和感を言語化するのが苦手だ。

**良い聞き方：**「普段の業務でこの画面をどう使いますか？実際にやってみてください」
→ 操作してもらう中で、迷い・戸惑い・非効率が自然に可視化される。

**悪い聞き方：**「この機能は便利ですか？」
→ 「はい」としか言いようがない。Yes/No質問は情報量が少ない。

**良い聞き方：**「この機能がなかった場合、今の業務はどうやっていますか？」
→ 現状との差分が浮き彫りになり、本当に必要な機能かどうかが見える。
:::

特に重要なのは、**言葉にならないフィードバック**だ。操作中に一瞬手が止まる、画面を行き来する、「えーと」と考え込む。これらは言語化されないが、UIの問題を示すシグナルだ。

### フィードバックが前提を覆す場合

フィードバックの中には、Phase 0〜2の前提を覆すものが含まれることがある。

例えば、プロトタイプを見た顧客が「実はこの業務は来期から廃止予定なんです」と言い出す。これはPhase 0の現状把握が不十分だったことを意味する。

こういう場合は、Phase 0〜2への差し戻しを躊躇してはいけない。プロトタイプまで作ったのにもったいない、と思うかもしれない。だが、廃止予定の業務のためのシステムを本実装まで進める方が、はるかにもったいない。

### 完了条件

- キーパーソン（意思決定者）からプロトタイプへの承認が得られている
- 未反映のフィードバックがある場合、対応方針が合意されている
- フィードバック反映後のI/F設計に矛盾がない

「キーパーソンの承認」が明示的に求められている点に注目してほしい。現場の担当者が「いいと思います」と言っても、意思決定権を持つ人の承認がなければ先に進まない。

## Phase 7：MVP構築

### MVPの範囲を限定する

MVP（Minimum Viable Product）に含める機能は、「直近でユーザーが使用する機能」に限定する。

ここでオペレーターが犯しがちな間違いは、「せっかくだからあれもこれも」と機能を追加してしまうことだ。MVPは最小限の実用品だ。将来必要になるかもしれない機能は入れない。

### 5つのAIロールが本格稼働する

Phase 7から、5つのAIロールが本格的に稼働を始める。

ここまでのフェーズでは、主にオペレーターとナビゲーター（壁打ち相手）の対話で進んでいた。Phase 7からはコーディングエージェントが実装を行い、コードレビュアーが品質レビューを行い、システム監査官が安全性を検証し、PM・スクラムマスターが進捗を管理し、ユーザー・運用サポートがテストシナリオを実行する。

この体制については、次章で詳しく解説する。

### 完了条件

- 直近でユーザーが使用する機能がすべて実装されている
- コードレビュアーの7視点レビューが完了し、オペレーターがサマリーを承認している（品質ゲート通過）
- システム監査官の監査からリリースOKの判断が出ている（安全ゲート通過）
- ユーザー・運用サポートのテストシナリオからリリースOKの判断が出ている

:::message alert
4つすべてがOKでなければMVPリリースを許可しない。「レビューは後でやる」「監査は運用開始後でいい」は認めない。
:::

## Phase 8：フルスケール実装

### 速度が上がっても品質プロセスを省略しない

MVPリリース後、残りの機能を実装する。5つのAIロールをフル稼働させ、高速に開発サイクルを回す。

ここで最も重要な注意事項は、**速度が上がっても品質プロセスを省略しないこと**だ。

:::message alert
速度が上がっても、コードレビュアーの7視点レビューとシステム監査官の監査、オペレーターによるサマリー確認を省略しない。MVPが動いている安心感から品質チェックが緩むフェーズでもある。
:::

Phase 8はMVPが動いている安心感から、品質チェックが緩みやすいフェーズでもある。「MVPはしっかりレビューしたから、追加機能は軽くでいいだろう」。この判断が品質劣化を招く。

プッシュごとにコードレビュアーの7視点レビューを実行し、システム監査官の監査を通す。オペレーターはサマリーを確認し、承認または差し戻しを判断する。この手順は省略しない。

### フル稼働体制

| ロール | 稼働内容 |
|--------|---------|
| PM・スクラムマスター | 進捗管理、要件との整合性確認、フィードバック共有 |
| コーディングエージェント | 実装の主体。オペレーターはレビュー結果の判断・承認に専念 |
| コードレビュアー | プッシュごとの7視点コード品質レビュー（品質ゲート） |
| システム監査官 | プッシュごとの監査（安全性・安定性・可用性）（安全ゲート） |
| ユーザー・運用サポート | テストシナリオの継続的実行、UX検証 |

### 差し戻す勇気

Phase 8で最も難しいのは、差し戻しの判断だ。

フルスケール実装を進めている最中に、ユーザー・運用サポートから「このフローだと実業務では使えない」というフィードバックが来ることがある。これがPhase 0〜2の前提を覆すものであれば、Phase 0〜2に差し戻す必要がある。

MVPリリース済みで、残り機能の実装も進んでいる状態で「Phase 2に戻ります」と言うのは勇気がいる。だが、使えないシステムを完成させることに意味はない。

この手法の最上位原則は「ユーザーの意図を完遂させる」だ。ユーザーの意図が完遂できないと判明した時点で、どのフェーズにいようと立ち戻る。それがこの手法の設計思想だ。

## フェーズ間の差し戻しルール

どのフェーズからでも、以下の条件に該当する場合は前のフェーズに差し戻す。

:::details フェーズ間の差し戻しルール表
| 検知した問題 | 差し戻し先 |
|-------------|-----------|
| 顧客の真の課題の認識が変わった | Phase 1 |
| 新しいユースケースが発見された | Phase 2 |
| 要件の追加・変更が必要 | Phase 3 |
| 技術スタックの変更が必要 | Phase 4 |
| I/F設計に重大な矛盾 | Phase 5 |
| 関係者の合意が崩れた | Phase 6 |
:::

差し戻し時に重要なのは、「なぜ戻るのか」と「何を解決して戻ってくるのか」を明確にすることだ。漫然と戻るのではなく、解決すべき課題を特定してから戻る。
